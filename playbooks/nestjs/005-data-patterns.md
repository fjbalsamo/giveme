# Guardrail 005 — Data patterns

## Context

Sequential or autoincremental IDs exposed to the client are a security
risk — they reveal business volume, enable enumeration attacks, and couple
the API contract to the database implementation. A client that knows
customer 42 exists will try customer 43.

Irreversible migrations are a production incident waiting to happen.
Entities without timestamps make debugging a guessing game. And optimistic
locking might sound like an edge case until two users edit the same record
at the same time and one of them silently loses their changes.

## Decision

All public IDs are UUID v4, generated by Prisma. Tables and columns follow
snake_case in the database, PascalCase in Prisma models. Every mutable
entity has a `version` column for optimistic locking. Every migration is
reversible. Business entity deletes are hard deletes — no soft delete
columns unless an explicit audit table is in place.

## Rules

- MUST: Public IDs are UUID v4 — `@default(uuid()) @db.Uuid` in Prisma schema
- MUST: Table names and column names in `snake_case` in the database — use `@@map` and `@map` in Prisma
- MUST: Prisma model names in `PascalCase` singular — `Customer`, not `customers` or `Customers`
- MUST: Every table has `createdAt` and `updatedAt` — `@default(now())` and `@updatedAt`
- MUST: Every mutable entity (receives PATCH or PUT) has a `version` column — integer, default 1
- MUST: Update operations receive the current `version` from the client, verify it matches DB, increment on success
- MUST: Version mismatch responds with `409 Conflict` — `type: .../errors/version-mismatch`
- MUST: All migrations are reversible — every migration has a documented or tested rollback
- MUST NOT: Expose autoincremental or sequential IDs to the client
- MUST NOT: Use soft delete columns (`deletedAt`, `isDeleted`) — use hard DELETE with audit tables if history is needed
- SHOULD: Use `@@map('table_name')` at the model level to make the translation explicit
- SHOULD: Document explicitly in `plan.md` when an entity is exempt from optimistic locking (read-only + delete only entities)

## Examples

### Correct

```prisma
// schema.prisma
model Customer {
  id        String   @id @default(uuid()) @db.Uuid
  name      String   @db.VarChar(100)
  email     String   @unique @db.VarChar(254)
  version   Int      @default(1)           // optimistic locking
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@map("customers")                       // snake_case in DB, PascalCase in Prisma
}
```

```typescript
// src/customers/customers.repository.ts
async update(id: string, dto: UpdateCustomerDto, version: number) {
  const updated = await this.prisma.customer.updateMany({
    where: {
      id,
      version,                             // verify version matches
    },
    data: {
      ...dto,
      version: { increment: 1 },          // increment on success
    },
  });

  if (updated.count === 0) {
    // either not found or version mismatch
    const exists = await this.prisma.customer.findUnique({ where: { id } });
    if (!exists) throw new NotFoundException(`Customer ${id} not found`);
    throw new ConflictException({
      type: 'https://your-service.com/errors/version-mismatch',
      title: 'Version mismatch',
      detail: 'The resource was modified by another request. Fetch the latest version and retry.',
    });
  }
}
```

### Incorrect

```prisma
// ❌ autoincremental ID exposed publicly
model Customer {
  id    Int    @id @default(autoincrement())  // use uuid() instead
  name  String
}

// ❌ no timestamps
model Customer {
  id   String @id @default(uuid()) @db.Uuid
  name String
  // missing createdAt and updatedAt
}

// ❌ soft delete
model Customer {
  id        String    @id @default(uuid()) @db.Uuid
  deletedAt DateTime?                           // use hard DELETE + audit table
}

// ❌ PascalCase in DB, no @@map
model customers {                               // model name should be Customer
  Id   String @id                              // column should be id with @map
}
```

```typescript
// ❌ update without optimistic locking
async update(id: string, dto: UpdateCustomerDto) {
  return this.prisma.customer.update({
    where: { id },                             // no version check — last write wins
    data: dto,
  });
}
```

## Verify signal

Reject if:
- A Prisma model uses `@default(autoincrement())` for a public-facing entity
- A Prisma model is missing `createdAt` or `updatedAt` fields
- A Prisma model has a field named `deletedAt` or `isDeleted` without a corresponding audit table documented in `plan.md`
- A `*.repository.ts` `update` or `updateMany` call has no `version` field in the `where` clause for entities marked as mutable in `plan.md`
- A Prisma model name is lowercase or plural (e.g. `customers`, `customer_record`)
- A Prisma model with more than 2 fields is missing `@@map()`

Flag for human review if:
- A `version` mismatch does not result in a `409 Conflict` response
- A migration file has no rollback documented in a comment or companion file
- An entity explicitly exempted from optimistic locking in `plan.md` receives a `@Patch()` or `@Put()` endpoint
